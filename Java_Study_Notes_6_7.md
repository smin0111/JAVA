## 개요

이 구간은 단순 자바 문법을 넘어 객체지향 프로그래밍(OOP)의 실질적인 뼈대를 잡는 핵심 파트다.
이 두 장을 이해하는 것은 자바 문법 사용자에서 객체지향 개발자로 넘어가는 중요한 관문이다.

# 6장. 클래스와 객체 (객체지향 시작)

## 1. 객체지향이 필요한 이유

기존 절차지향 프로그래밍 방식은 코드가 커질수록 꼬이기 쉽고, 재사용이나 유지보수가 너무 힘들었다.
현실 세계의 사물 단위처럼 객체 단위로 프로그램을 구성하여 코드 모듈화와 재사용성을 극대화하기 위해 등장했다.

## 2. 클래스 vs 객체

### 클래스
무언가를 만들어내기 위한 설계도, 틀, 템플릿이다. (예: 자동차 설계도)

### 객체
클래스라는 틀을 기반으로 메모리상에 실제로 찍혀나온 결과물이다. 인스턴스라고도 부른다. (예: 도면을 보고 만든 실제 자동차)

```java
// 클래스(설계도) 정의
class Car {
    String color;
    int speed;
}

// 객체(실제 결과물) 생성
Car c = new Car();
```

간결하게 정리하면 클래스는 타입이고 객체는 실제 메모리에 뜬 실체이다.

---

## 3. 객체의 구성 요소

객체는 크게 두 가지 성질을 갖는다.

| 구성 | 의미 |
| --- | --- |
| 변수 | 상태, 속성 (예: 색깔, 현재 속도) |
| 메서드 | 동작, 행동 (예: 엑셀 밟기, 브레이크 밟기) |

```java
class Car {
    String color;     // 상태
    void drive(){}    // 행동
}
```

---

## 4. 변수 종류 (중요)

### 1) 인스턴스 변수
가장 기본이 되는 변수. 객체가 만들어질 때마다 객체별로 따로 생성되는 종속 변수다.

```java
class Car {
    String color;
} // 아반떼는 파랑, 소나타는 검정이듯 각 객체마다 다를 수 있다.
```

### 2) 클래스 변수 (static)
모든 객체가 공동으로 하나의 똑같은 값을 공유할 때 쓴다. static 키워드를 붙인다.

```java
static int wheelCount = 4; // 모든 차는 바퀴가 4개니까 값을 서로 공유함 (메모리 절약)
```

### 3) 지역 변수
메서드 안에서만 잠깐 살다가 끝나면 깔끔하게 지워지는 변수다.

---

## 5. 메서드

C언어 등의 함수 개념과 본질은 같지만, 객체 내부에 필수로 종속되어 있다는 철학적 차이가 있다.

```java
int add(int a, int b){
    return a + b;
}
```

주로 반환 타입, 메서드 이름, 매개변수, return 문으로 구성된다.

---

## 6. 메서드 호출 흐름

자바에서 내부적으로 이뤄지는 데이터 메모리 구조의 핵심은 Call Stack(호출 스택)이다.
메서드를 실행하면 박스 위에 박스를 쌓듯이 스택에 임시로 쌓여 올라간다(Push).
제일 위에 존재하는 메서드가 끝나면 그 즉시 제거되며(Pop), 자신을 불렀던 이전 층으로 알아서 되돌아간다.

---

## 7. 기본형 vs 참조형 (중요)

### 기본형 (Primitive Type)
말 그대로 진짜 숫자(값) 자체를 변수에 저장한다. int, double, boolean 등.

### 참조형 (Reference Type)
그 객체가 존재하는 실제 메모리상의 번지수 주소값을 저장한다.

```java
Car c1 = new Car(); 
Car c2 = c1;
```

이렇게 하면 c1과 c2 둘 다 같은 주소를 보고 있게 되므로, c2의 색상을 바꾸면 c1의 색상도 바뀐다 (동일 객체).

---

## 8. 생성자 (Constructor)

객체가 new 연산자를 만나 처음 탄생할 때 무조건 단 한 번 자동 실행되는 초기화 메서드다.

```java
class Car {
    Car() {
        System.out.println("생성 완료");
    }
}
```

특징:
- 이름이 무조건 클래스 이름과 똑같아야 한다.
- 반환 타입 자체가 애초에 없다. void 조차도 안 적는다.
- 내 의지와 상관없이 객체가 만들어지는 즉시 자동 호출된다.

---

## 9. 생성자 오버로딩

매개변수의 갯수나 타입을 다르게 해서 객체를 만드는 선택지를 다양하게 늘릴 수 있다.

```java
Car(){} // 옵션 없음
Car(String color){} // 컬러만 내가 고름
Car(String color, int speed){} // 풀옵션 선택
```

---

## 10. this 키워드

메서드 코드 안에서 현재 자기 자신 객체의 주소를 가리킬 때 쓰는 키워드다.

```java
class Car {
    String color;

    Car(String color){ // 매개변수
        this.color = color; // 내 속성(color) = 니가 넘긴 값(color)
    }
}
```

매개변수 이름과 내 변수 이름이 겹쳤을 때, 내 인스턴스 변수임을 똑똑히 증명하기 위해 강제로 붙인다.

---

## 11. this() 생성자 호출

생성자 안에서 너무 중복되는 코드를 치기 싫을 때, 내 클래스의 또 다른 버전의 생성자를 재활용 호출할 때 쓴다.

```java
Car(){
    this("red"); // 난 기본값으로 red를 처리하는 생성자를 부를게!
}
```

규칙: 실행 순서가 꼬이는 걸 막기 위해 무조건 생성자의 가장 첫 줄에만 쓸 수 있다.

---

## 12. 메서드 오버로딩 (Overloading)

똑같은 이름을 가진 메서드를 여러 개 만들어 두는 것이다.

조건:
- 이름은 무조건 같아야 함.
- 자바가 구별할 수 있어야 하니 매개변수 갯수나 타입이 서로 달라야 함.

```java
int add(int a, int b) { ... }
double add(double a, double b) { ... }
```

---

## 13. 캡슐화 (Encapsulation)

무방비로 객체가 오염되는 것을 막고자 객체의 내부 내용을 보호막으로 싸맨다는 개념이다.

필요성:
외부 사람이 내 클래스의 민감 변수를 직접 맘대로 손대는 것을 막기 위해서다.

```java
private int age; // 외부 접근 완전 차단

// 합법적으로 허가된 통로(메서드)만 열어줌
void setAge(int age){
    if(age >= 0) this.age = age; // 타당한 값인지 여기서 방어 가능
}
```

데이터 무결성을 보호하는 최선봉 기술이다.

# 7장. 상속과 다형성

객체지향다운 맛을 내는 장.

---

## 1. 상속 (Inheritance)

처음부터 100% 다 짜는 게 아니라 유용한 기존 클래스의 틀을 물려받고 확장하여 새 클래스를 만드는 것이다. 

```java
class Child extends Parent {} // 난 부모 거 다 내 걸로 쓰면서 시작한다
```

## 2. 궁극적 이유

상속이 없으면 중복 코드가 엄청나게 늘어난다.
부모 쪽에 공통 기능을 한 번 짜두고 자식들은 그 코드를 물려받아 재사용성의 이점을 챙긴다.

## 3. 상속 특징

자식은 부모가 물려준 변수와 메서드를 내 것처럼 쓸 수 있다. private 같은 제어자로 막은 건 빼고.

## 4. 상속 관계 개념

자식은 기능이 항상 더 추가되는 형태이므로, 부모보다 더 구체적이고 덩치가 크다.
Person 보다 Student 가 더 많은 속성이나 기능을 가짐. 반대로는 성립 안 함.

---

## 5. 메서드 오버라이딩 (Overriding)

부모에게 상속받은 메서드가 맘에 안 들 때, 본인 코드 블록에서 내 입맛대로 재정의해서 덮어쓰는 문법이다.

```java
class Parent{
    void run(){}
}

class Child extends Parent{
    @Override
    void run(){ /* 나만의 방식으로 달림 */ }
}
```

오버라이딩 3대 조건:
부모 메서드와 다음 조건이 쌍둥이처럼 똑같아야 인정된다.
1. 이름이 같을 것
2. 매개변수가 같을 것
3. 반환 타입이 무조건 같을 것

---

## 6. super 키워드

자식 객체 내부에서, 오버라이딩에 가려진 진짜 오리지널 부모 측의 속성이나 기능을 다시 한 번 모셔올 때 누르는 리모컨이다.

```java
super.run(); // 자식꺼 말고 부모 원래 거 실행해주세요!
```

---

## 7. 생성자와 상속

자식 객체가 태어나기 전에는 무조건 뼈대를 잡는 부모 객체부터 순서대로 먼저 태어나야 정상 구동이 된다.
따라서 객체가 만들어질 때 자식 생성자 내부에서 부모 생성자가 무조건적으로 먼저 호출된다.

super() 생성자 호출:
부모 생성자를 강제로 집어 호출할 때 쓴다.

```java
Student(){
    super("부모님 나이 세팅!");
}
```

this() 처럼 얘도 무조건 첫 줄에 적어야 한다.

# 8. 다형성 (Polymorphism) - 초핵심 심화 부분

## 다형성이란?
단순하게 말해 하나의 부모 타입 변수로 여러 종류의 자식 객체들을 통일해서 다루는 것이다.
프로젝트에서 결제(Payment) 시스템이나 AI 인텐트(의도) 처리기를 만들 때 가장 핵심이 되는 원칙이다.

```java
// 부모 타입 변수 = 자식 객체
Payment p = new KakaoPay();
```
실제로 생성된 객체는 카카오페이(자식)지만, 겉으로 다룰 때는 상위 개념인 '결제수단(부모)' 타입으로 취급하겠다는 뜻이다.

## 이걸 왜 쓸까? (다형성의 극강 목적)
구현 방식이 서로 다른 여러 객체들이라도 결국 같은 부모(예: Payment)로 묶인다면, 코드 작성 시 단 하나의 배열이나 메서드 매개변수로 묶어서 아주 일관되고 유연하게 관리할 수 있기 때문이다.

```java
// 내가 진행한 프로젝트 결제 모듈 유사 예시
Payment[] payMethods = new Payment[3];
payMethods[0] = new KakaoPay();
payMethods[1] = new TossPay();
payMethods[2] = new NaverPay(); // 전부 부모인 Payment 배열 하나로 통제 끝

// 결제 종류가 뭐든 간에 부모에 있는 공통 메서드 하나로 전부 일괄 처리 가능
for(Payment p : payMethods) {
    p.processPay(); 
}
```
나중에 새로운 결제 수단(ApplePay)이 추가되어도 핵심 반복문 로직은 단 1줄도 수정할 필요가 없다. (유지보수와 확장에 극강)

## 다형성의 핵심 규칙과 한계
부모 타입(Payment)으로 묶어서 관리하다 보니, 자바 입장에선 이 객체의 진짜 속내가 카카오페이인지 토스페이인지 당장은 모른다. 따라서 부모 클래스(Payment)에 선언되어 있는 공통 기능만 일단 안전하게 호출을 허용한다.

### 자식의 고유 기능을 꼭 써야 한다면?
카카오페이만 가지고 있는 특수 기능(예: 카톡 알림톡 전송 메서드)을 써야 한다면, 원래 자기 신분증(KakaoPay)으로 임시 복귀하게 해주는 다운캐스팅 신공을 써야 한다.

```java
KakaoPay kakao = (KakaoPay) p; // 강제 다운캐스팅 형변환
kakao.sendKakaoMsg(); // 본인 고유 기능 제약 해제 허용
```

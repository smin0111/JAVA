# 자바의 정석 4판 13장 - 쓰레드(Thread)

# 1. 프로세스 vs 쓰레드 (무조건 나오는 기본)

## 프로세스(Process)
실행 중인 프로그램 하나

예)
크롬 실행 -> 프로세스 1개
게임 실행 -> 프로세스 1개

프로세스 = 철저히 독립된 메모리 공간

---

## 쓰레드(Thread)
프로세스 안에서 실행되는 작업 단위

한 프로그램 안에서 동시에 여러 작업 가능하게 해줌.

예)
크롬 안에서
- 다운로드
- 영상 재생
- 페이지 로딩
이게 전부 쓰레드다.

### 핵심 정리

| 구분 | 프로세스 | 쓰레드 |
|---|---|---|
| 메모리 | 독립 | 공유 |
| 생성비용 | 큼 | 작음 |
| 속도 | 느림 | 빠름 |

그래서 멀티쓰레드가 중요하다.

---

# 2. 멀티쓰레드란?

한 프로그램이 여러 작업을 동시에 수행하는 것이다.

예)
- 게임 + 음악 + 채팅
- 서버: 여러 사용자 동시 처리

서버 개발 = 100% 멀티쓰레드 환경이다.

---

# 3. 쓰레드 만드는 방법 (시험/면접 단골)

자바에서 2가지 방법이 있다.

## 방법 1. Thread 클래스 상속

```java
// Thread 라는 자바 기본 클래스를 부모로 삼아 기능들을 상속받는다
class MyThread extends Thread {
    // Thread가 가진 run() 메서드를 오버라이딩하여 내가 할 작업을 채워넣는다
    public void run() {
        // 독립된 공간에서 쓰레드가 실제로 행동할 코드를 이 안에 작성한다
        System.out.println("쓰레드 실행!");
    }
}

public class Main {
    public static void main(String[] args) {
        // 내가 만든 클래스를 객체로 복사해서 찍어낸다
        MyThread t = new MyThread();
        
        // 주의: t.run()을 직접 부르지 않고, t.start()를 호출한다
        t.start();
    }
}
```

중요:
run()은 직접 호출 X
start() 사용 O

왜?
start() -> 운영체제에 새 쓰레드 전용 작업 공간(스택) 생성 요청 후 그 위에서 돌림
run() -> 새로운 흐름 없이, 그냥 원래 있던 메인 쓰레드 위에서 평범한 메서드로 호출됨 (동시성 0%)

시험 무조건 나옴.

---

## 방법 2. Runnable 인터페이스 (실무 표준)

```java
// 부모 클래스를 상속받는 대신 Runnable 이라는 껍데기 인터페이스 틀을 구현한다
class MyRunnable implements Runnable {
    // Runnable이 강제하는 run() 메서드를 오버라이딩한다
    public void run() {
        System.out.println("쓰레드 실행!");
    }
}

public class Main {
    public static void main(String[] args) {
        // 1. 내가 만든 작업 내용이 담긴 객체(MyRunnable)를 먼저 조립한다
        // 2. 진짜 쓰레드 객체(Thread)를 만들 때, 부품처럼 그 작업 객체를 던져넣는다
        Thread t = new Thread(new MyRunnable());
        
        // 3. 실행 방법은 위와 똑같이 start()를 호출한다
        t.start();
    }
}

// Blue 백엔드 실무 적용 예시 (비동기 알림 처리):
// 회원이 예약이나 결제를 성공했을 때 알림톡/메일을 발송하는 로직이 3~5초 지연된다고 가정하자.
// 메인 쓰레드가 그걸 다 기다려주면 유저는 사이트가 멈춘 줄 알고 새로고침을 누르게 된다.
// 이럴 때 Runnable 쓰레드로 분리해서 던지면 메인 쓰레드는 즉시 200 OK를 내려줄 수 있다.

// Java 8 람다식 Runnable 적용 (스프링 부트 환경)
public void processReservation(ReservationDTO dto) {
    // 1. 핵심 비즈니스 로직 (0.1초 소요)
    reservationMapper.insertReservation(dto);
    
    // 2. 알림 발송 로직은 Runnable 작업으로 빼서 백그라운드 쓰레드에 넘겨버린다
    Runnable sendMailTask = () -> {
        System.out.println(dto.getMemberId() + "님에게 예약 완료 알림톡 발송 (3초 소요...)");
        // 실제로는 mailService.send() 호출
    };
    
    // 새로운 쓰레드 파생하여 시작
    new Thread(sendMailTask).start();
    
    // 3. 메인 쓰레드는 기다리지 않고 즉시 응답 반환
    System.out.println("예약이 완료되었습니다! (프론트로 200 OK 즉시 전송)");
}
```

왜 Runnable을 더 많이 쓰나?
자바는 단일 상속만 지원하므로, Thread를 상속하면 다른 중요한 클래스를 아예 상속할 수 없게 된다.
그래서 실무에서는 족보를 낭비하지 않는 Runnable이 무조건 표준이다.

# 4. 쓰레드 실행 흐름

```java
Thread t = new Thread();
t.start();
```

실행 단계:
1. NEW -> 시작 전 객체만 단순히 생성된 상태
2. RUNNABLE -> 실행 대기열(Queue) 안에서 서로 CPU를 차지하려고 줄 서 있는 상태
3. RUNNING -> 운 좋게 CPU를 배정받아 실제로 코드가 실행 중인 상태
4. TERMINATED -> 일이 다 끝나고 수명이 다해 소멸한 상태

# 5. sleep() - 쓰레드 잠재우기

```java
// 매개변수는 밀리초(ms) 단위라서 1000이 1초다
Thread.sleep(1000);
```

1초 정지시킨다.

주의 사항:
```java
try {
    // 자고 있는 도중에 밖에서 흔들어 깨우면(interrupt), 강제로 잠이 깨며 이 안에서 InterruptedException이 터진다
    // 그래서 컴파일러가 try-catch를 안 쓰면 아예 실행을 막아버리므로 필수적으로 작성해야 한다 
    Thread.sleep(1000);
} catch (InterruptedException e) {
    // 깨어났을 때 대처할 로직
}
```

예외 발생 가능 메서드이므로 시험 단골이다.

---

# 6. interrupt() - 쓰레드 깨우기

sleep 중인 쓰레드를 강제로 깨운다.

```java
// 대상 쓰레드(t)에게 "일어나라"고 예외(Exception) 신호를 날린다
t.interrupt();
```

InterruptedException이 발생하며 자던 쓰레드가 catch 블록으로 이동해서 깨어난다.

---

# 7. join() - 쓰레드 끝날 때까지 기다림

```java
// 이 코드를 만난 메인 쓰레드는, 대상 쓰레드인 t가 완전히 끝날 때까지 블로킹(대기)되어 하염없이 기다린다
t.join();
```

메인 쓰레드가 기다림.

예)
- 파일 다운로드 끝날 때까지 대기
- 모든 데이터 처리가 완료된 후 한방에 결과 합쳐서 출력할 때 사용

---

# 8. yield() - 양보하기

```java
// 자기가 돌고 있던 CPU의 점유 시간을, 줄 서 대기하고 있는 다른 쓰레드에게 넘긴다
Thread.yield();
```

다른 쓰레드에게 CPU 양보.
거의 안 쓴다 (시험/이론용 개념).

// 1에서 10까지 숫자를 부여해 10이면 제일 먼저 실행되게끔 시도한다

# 10. 데몬 쓰레드 (Daemon Thread)

일반적인 메인 작업을 돕는 보조 쓰레드다.

특징: 메인 쓰레드가 끝나면 같이 강제 종료된다 (순장 당함).

대표 예:

# 자바의 정석 4판 13장 - 동기화 파트

# 1. 멀티쓰레드의 진짜 문제

멀티쓰레드는 빠르고 좋지만 큰 문제 1개가 발생한다.

공유자원 문제 (Race Condition)
여러 쓰레드가 같은 변수에 동시에 접근하는 문제다.

예제: 은행 계좌
쓰레드 A -> 출금 버튼 누름
쓰레드 B -> 출금 버튼 누름

동시에 실행되면 잔액 계산이 꼬여버린다.

코드로 보면:
```java
class Account {
    int balance = 1000;

    void withdraw(int money) {
        // 만약 통장 잔액이 요구 금액보다 크거나 같으면? 이라는 조건이다
        if (balance >= money) {
            // [위험 지점] 쓰레드 A가 위 조건을 통과한 직후 타이밍 좋게 멈췄다
            // 곧바로 쓰레드 B가 들어와서 위 조건을 똑같이 통과해버렸다 
            // 둘 다 if문 안쪽으로 진입 성공

            // 결국 A도 빼가고 B도 빼가서 1000원이 두 번 증발한다 
            balance -= money;
        }
    }
}
```

결과: 잔액 = -1000 원이라는 대참사가 발생.
이걸 Race Condition 이라고 한다.

# 2. 해결책 = 동기화 (Synchronization)

한 번에 한 쓰레드만 접근하게 만들기.
화장실 문 잠금과 똑같은 개념이다.

---

# 3. synchronized 키워드 (시험 단골)

자바 동기화의 핵심 키워드다.

## 방법 1. 메서드 전체 잠금
```java
// 반환 타입 앞에 키워드를 박으면, 이 함수에 누군가 들어가는 순간 이 객체의 열쇠가 잠겨버린다
synchronized void withdraw(int money) {
    // 밖에서 다른 쓰레드들이 줄 서서 기다리는 동안, 락을 쥔 쓰레드만 혼자 안전하게 모든 로직을 처리한다
    if (balance >= money) {
        balance -= money;
    }
}
```
이 메서드는 한 번에 하나의 쓰레드만 실행 가능하다.

## 방법 2. 특정 코드 블럭만 잠금
```java
void withdraw(int money) {
    // this(이 통장 객체)의 락(Lock)을 획득해야 괄호 안으로 들어갈 수 있다
    synchronized(this) {
        if (balance >= money) {
            balance -= money;
        }
    }
}
```
더 많이 쓰고 권장되는 방식이다. (전체를 잠그면 대기줄이 기하급수적으로 길어져 서버가 죽을 수 있기 때문)

핵심 개념 = 객체 Lock
객체 하나당 Lock(열쇠)이 딱 1개씩 존재한다. 
객체 = 화장실, Lock = 열쇠. 한 쓰레드가 들어가면 다른 쓰레드는 밖에서 하염없이 기다려야 한다.

Blue 백엔드 예시 (동시 결제 및 좌석 예약 차감 방어):
토스 페이먼츠 결제 승인 확인 로직이나 영화관 좌석 예약 로직에서 중복 처리가 발생하는 걸 막을 때 synchronized를 적용할 수 있다. 

```java
public class ReservationService {
    // 좌석 남은 개수
    private int availableSeat = 1;
    
    // 이 객체 락을 얻은 딱 하나의 쓰레드(요청)만 좌석을 예약할 수 있다.
    // 만약 A와 B 유저가 0.001초 차이로 동시 클릭했다면?
    public void executeReservation(String memberId) {
        synchronized(this) {
            // 안으로 들어온 1번 쓰레드가 자리를 확인하고 선점한다.
            if(availableSeat > 0) {
                System.out.println(memberId + " 예약 성공!");
                availableSeat--; // 좌석 소모
            } else {
                // 밖에서 줄 서서 기다리던 2번 쓰레드가 뒤늦게 들어오면 
                // 이미 availableSeat은 0이 되어있어 예약이 실패한다 (데이터 보호 성공)
                throw new BusinessException("이미 매진된 좌석입니다.");
            }
        }
    }
}
```
참고 실무 꿀팁: 
사실 자바의 `synchronized`는 톰캣 서버가 딱 1대일 때만 작동한다. 만약 Blue-back 서버를 2대 3대로 늘려서 서비스하면, 자바 안의 열쇠(Lock)가 각각 따로 존재하므로 막을 수 없다. 그래서 실무(여러 대의 서버)에서는 DB의 Lock 기능이나 Redis 같은 더 수준 높은 동기화 기술을 써서 서버들을 하나로 통제한다. (하지만 그 근본 원리는 이 동기화 개념과 똑같다!)

# 4. 동기화의 문제점

동기화 = 데이터는 안전하지만 속도는 느리다.
왜? 밖에서 줄 서서 자기 차례를 기다려야 하기 때문이다.
그래서 최소한의 줄 수만 블록으로 동기화해야 한다.

# 5. wait() / notify() (진짜 핵심)

쓰레드들이 서로 돕기 위한 협력 기술이다.

- wait() -> 기다림
- notify() -> 깨움

Producer(생산) / Consumer(소비) 패턴에 쓰인다.

## 빵집 예제 (시험 단골)
상황: 
생산자: 빵을 만듦
소비자: 빵을 먹음

빵이 없으면 어떻게 되나?
소비자는 빵이 채워질 때까지 기다려야 한다.

### wait()
```java
// 쓰레드가 혼자 남들 진입도 못하게 열쇠만 쥔 채로 버티지 않고
// 열쇠를 바닥에 내려놓아 남들이 들어올 수 있게 양보한 뒤, 자신은 대기실에서 대기한다
wait();
```
중요: synchronized 안에서만 유효하고 사용 가능한 메서드다.

### notify()
```java
// 빵집 주인이 빵을 만들고 진열대(공유자원)에 올린 뒤
// 대기실에서 잠자고 대기하던 쓰레드 중 하나를 깨워서 다시 일하게 만드는 신호 알림 메서드다
notify();
```

흐름 이해 (가장 중요):
1. 쓰레드 A(소비자) -> 통과했는데 빵이 없어서 wait() 실행 -> Lock을 바닥에 떨구고 대기실에서 잠듦
2. 쓰레드 B(생산자) -> 열쇠 주워서 진입 후 빵 만들고 작업 완료 -> 자고 있는 A 들으라고 notify() 외침
3. 쓰레드 A -> 소리 듣고 잠에서 깨어남 -> 빵 집어 먹고 행복하게 떠남

# 6. sleep vs wait 차이 (시험 무조건)

가장 무섭고 중요한 단골 질문.

| 구분 | sleep() | wait() |
|---|---|---|
| Lock 반납 | 안함 (열쇠를 주머니에 꼭 쥔 채 졸음) | 함 (열쇠를 착하게 반납하고 양보함) |
| 위치 | 아무데나 자유롭게 사용 가능 | 오직 synchronized(동기화 영역) 안에서만 |
| 목적 | 단순 자기 시간 멈추기 | 쓰레드 간 자원 공유 및 협력을 위한 대기 |

# 7. notify vs notifyAll

```java
notify();     // 랜덤으로 하나만 깨움
notifyAll();  // 자는 쓰레드를 모두 다 기상시킴
```

실무에서는 notifyAll 권장이다. 
왜? notify는 무작위성을 띄어서 당장 일이 없는 엉뚱한 쓰레드가 깨어나버리면 또 꼬이기 때문이다.

# 8. 교착상태 (Deadlock)

정말 끔찍한 쓰레드 지옥이다.

상황:
쓰레드 A -> 열쇠 1(Lock1)을 잡고 열쇠 2(Lock2)를 얻으려고 기다림. 
쓰레드 B -> 열쇠 2(Lock2)를 이미 잡고 열쇠 1(Lock1)을 얻으려고 기다림.

둘 다 서로가 가진 것을 포기하지 않고 영원히 상대방을 기다리는 상태다. 
해결 방법:
- 자물쇠를 따는 순서를 항상 한 가지 방향으로 통일한다.
- 락 중첩 등 무의미한 동기화를 과감히 최소화한다.
